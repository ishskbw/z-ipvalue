# SMK 자동 생성 기능 설계 문서

> **작성 목적**: 특허번호 입력 시 AI가 기술소개서(SMK)를 자동 생성하고, 관리자 검토 후 플랫폼에 게시하는 기능의 설계 명세.
>
> **작업 방식**: 이 문서를 기반으로 Claude Code가 단계별로 구현.
>
> **참조 레퍼런스**: KISTA(한국특허전략개발원) 2024 융합연구 권리·사업화 지원사업 기술소개서 포맷.

---

## 1. 서비스 컨텍스트

- **사이트**: z-ipvalue.com (특허 기술거래 플랫폼)
- **주요 특허 출처**: 대학, 공공연구기관, 병원 (기초·응용 연구 성격)
- **운영 주체**: 제니스국제특허법률사무소
- **타겟 사용자**:
  - **등록자 측**: 대학 산학협력단, 공공연 TLO, 병원 기술이전센터
  - **열람자 측**: 기술 도입 검토 기업, 투자자, R&D 기획자
- **AI 포지셔닝**: 사용자에게는 AI 언급 최소화, 제니스 담당자 검토를 신뢰 요소로 강조

## 2. 항목 분류

### 2.1 AI 자동 생성 항목 (명세서 + 서지정보 기반)

| # | 항목 | 분량 | 설명 |
|---|---|---|---|
| 1 | **기술 개요** | 2~3문장 | 이 기술이 무엇을 하는지, 어디에 쓰이는지 압축 요약 |
| 2 | **기술개발 배경** | 1~3문장 | 해결하고자 하는 문제, 기존 기술의 한계 |
| 3 | **핵심 기술 내용** | 3~5개 불릿 | 기술의 작동 원리, 구성 요소, 구현 방법 |
| 4 | **특징 및 장점** | 3~5개 불릿 | 기존 대비 차별점, 경쟁 우위 |
| 5 | **활용 분야** | 2~4개 불릿 | 적용 가능한 산업·제품·서비스 영역 |
| 6 | **기대 효과** | 2~4개 불릿 | 기술 도입 시 얻을 수 있는 경제적·사회적 가치 |
| 7 | **핵심 키워드** | 5~8개 태그 | 검색·분류용 키워드 |

### 2.2 KIPRIS 자동 연동 항목 (이미 구현)

- 특허번호 (출원/공개/등록)
- 출원일, 등록일, 공개번호
- 출원인, 발명자
- 심사 현황
- 해외 출원 현황 (패밀리 특허)
- 대표 도면

### 2.3 수동 입력 항목 (담당자가 채움)

| 항목 | 이유 |
|---|---|
| **TRL 단계** (1~9) | 실제 연구 진행 상황은 연구자/담당자만 앎 |
| **시장 동향** (시장 규모, CAGR, 출처) | AI 환각 리스크 — 숫자 신뢰성 확보 |
| **거래 형태** (라이선스/매각/공동개발) | 권리자와의 협의 결과 |
| **희망가** (협의/제시가) | 협상 정보 |
| **추가 참고 이미지** | 연구자 제공 자료, 도면 외 시각물 |
| **등록 상태** (비공개/공개/거래중/거래완료) | 플랫폼 운영 상태 |

## 3. 데이터베이스 설계 (Supabase `patents` 테이블)

### 3.1 기존 스키마 유지

현재 컬럼 파악 후 유지 항목 분류 필요 (Code에게 첫 작업으로 `lib/supabase.js` 와 실제 테이블 스키마 조사 요청).

### 3.2 추가 필요 컬럼 (제안)

```sql
-- AI 생성 콘텐츠 (JSON 또는 개별 컬럼)
tech_summary          TEXT          -- 기술 개요
tech_background       TEXT          -- 기술개발 배경
tech_core_content     TEXT[]        -- 핵심 기술 내용 (불릿 배열)
tech_features         TEXT[]        -- 특징 및 장점 (불릿 배열)
tech_applications     TEXT[]        -- 활용 분야 (불릿 배열)
tech_impact           TEXT[]        -- 기대 효과 (불릿 배열)
tech_keywords         TEXT[]        -- 핵심 키워드 (태그 배열)

-- 수동 입력 콘텐츠
trl_level             INT           -- 1~9
market_trend          TEXT          -- 시장 동향 (Markdown 허용)
market_size_usd       NUMERIC       -- 시장 규모 (선택)
market_cagr           NUMERIC       -- 연평균 성장률 (선택)
market_source         TEXT          -- 시장 데이터 출처
deal_type             TEXT          -- 거래 형태 (enum: 'license', 'sale', 'co_dev')
deal_price            TEXT          -- 희망가 (문자열 — '협의' 허용)
custom_images         TEXT[]        -- 추가 이미지 URL 배열

-- 상태 관리
status                TEXT          -- 'draft', 'ai_generated', 'reviewed', 'published', 'in_deal', 'completed'
ai_generated_at       TIMESTAMPTZ   -- AI 생성 시각
reviewed_by           UUID          -- 검토한 관리자
reviewed_at           TIMESTAMPTZ   -- 검토 완료 시각
published_at          TIMESTAMPTZ   -- 공개 시각
```

### 3.3 구현 시 결정 필요

- **배열 vs JSON**: `TEXT[]` 대신 `JSONB` 단일 컬럼으로 통합할지 검토
- **변경 이력**: AI 초기 생성본과 관리자 수정본을 비교 저장할지 (신뢰성 감사용)

## 4. AI 프롬프트 설계

### 4.1 입력

1. **KIPRIS 서지정보** (이미 구현된 API 경로)
2. **명세서 전문 XML** (KIPRIS에서 가져옴)
3. **도면 설명 텍스트** (`figureDescsFromXml` 파싱 결과)

### 4.2 시스템 프롬프트 (초안)

```
당신은 한국 특허 기술의 사업화 가능성을 평가하고 정리하는 전문 분석가입니다.
대학, 공공연구기관, 병원에서 출원한 특허를 기반으로, 기술이전 담당자와
도입 검토 기업이 한눈에 이해할 수 있는 기술소개서(SMK)를 작성합니다.

작성 원칙:
1. 사실 기반: 명세서에 명시된 내용만 사용. 추측·확장 금지.
2. 정량 데이터 금지: 시장 규모, 매출 전망 등 숫자는 일절 생성하지 말 것.
3. 간결성: 각 항목은 지정된 분량 엄수.
4. 객관적 톤: 과장·홍보성 표현 지양. "혁신적인", "최고의" 같은 수식어 배제.
5. 전문 용어 균형: 특허 분야 전문 용어는 유지하되, 처음 등장 시 간략 풀이 권장.
6. 불릿 형식: 지정된 항목은 반드시 불릿 배열로 반환.
7. 언어: 한국어, 공식 문서체 (~합니다, ~됩니다).

출력 형식: 반드시 다음 JSON 구조로만 응답. 전후 설명 없이 JSON만.
```

### 4.3 출력 스키마 (JSON)

```json
{
  "tech_summary": "string (2~3문장)",
  "tech_background": "string (1~3문장)",
  "tech_core_content": ["string", "string", "..."],
  "tech_features": ["string", "string", "..."],
  "tech_applications": ["string", "string", "..."],
  "tech_impact": ["string", "string", "..."],
  "tech_keywords": ["string", "string", "..."]
}
```

### 4.4 사용자 프롬프트 템플릿

```
다음 특허 정보를 바탕으로 SMK를 생성해주세요.

[특허 기본 정보]
- 특허명: {invention_title}
- 출원번호: {application_number}
- 출원인: {applicant}
- 발명자: {inventors}
- IPC 분류: {ipc_classification}

[명세서 전문]
요약: {abstract}
청구항: {claims}
발명의 상세한 설명: {detailed_description}

[도면 설명]
{figure_descriptions}

위 정보만으로 지정된 JSON 구조로 응답해주세요.
```

### 4.5 호출 정책

- **모델**: `claude-sonnet-4-20250514` 또는 최신 안정 모델 (비용·품질 균형)
- **max_tokens**: 3000 (JSON 구조 완성 보장)
- **temperature**: 0.3 (사실 기반 유지)
- **호출 위치**: **서버 사이드 전용** (`app/api/kipris/search/route.js` 확장 또는 신규 `app/api/smk/generate/route.js`)
- **키 관리**: `process.env.ANTHROPIC_API_KEY` (Vercel 환경변수)
- **실패 처리**: API 실패 시 빈 항목 반환하고 UI에서 "AI 생성 실패 — 수동 입력 필요" 배지 표시

## 5. 관리자 워크플로우

### 5.1 등록 플로우 (단계별)

```
[1단계] 특허번호 입력
   ↓
[2단계] KIPRIS 조회 (자동) → 서지정보·명세서·도면 수집
   ↓
[3단계] AI 생성 (자동 또는 버튼 클릭)
   - 7개 항목 (2.1 참조) AI가 생성
   - DB에 status='ai_generated' 로 저장
   ↓
[4단계] 관리자 검토 페이지 (핵심)
   - 각 항목별 AI 생성본 표시
   - 편집 가능 (수정·추가·삭제)
   - 수동 입력 항목 (TRL, 시장 동향 등) 채움
   - "검토 완료" 버튼 → status='reviewed'
   ↓
[5단계] 게시 버튼
   - "플랫폼 등록" → status='published'
   - 공개 페이지에서 검색 가능
```

### 5.2 관리자 검토 UI 요구사항

- **병렬 배치**: 좌측 "AI 원본", 우측 "수정본" 비교 뷰
- **변경 표시**: AI 원본과 달라진 부분 하이라이트
- **인라인 편집**: 각 항목 클릭 시 바로 편집 가능
- **자동 저장**: 편집 중 3초 유휴 시 draft 자동 저장
- **재생성 버튼**: "AI로 다시 생성" 항목별 가능 (수동 수정 내용 덮어쓰기 경고)
- **검토 이력**: 누가 언제 검토했는지 기록

## 6. 공개 페이지 레이아웃

### 6.1 섹션 배치 (KISTA 포맷 참조 + 웹 친화 조정)

```
┌─────────────────────────────────────────────┐
│ [헤더] 특허명 / 거래 형태 배지 / 희망가          │
├─────────────────────────────────────────────┤
│ [메타] 특허번호, 출원일, 출원인, 발명자, TRL    │
├─────────────────────────────────────────────┤
│ [기술 개요]  ← AI 생성                        │
├─────────────────────────────────────────────┤
│ [대표 도면] ← KIPRIS                          │
├─────────────────────────────────────────────┤
│ [기술개발 배경] ← AI 생성                     │
├─────────────────────────────────────────────┤
│ [핵심 기술 내용] ← AI 생성 (불릿)             │
├─────────────────────────────────────────────┤
│ [특징 및 장점] ← AI 생성 (불릿)               │
├─────────────────────────────────────────────┤
│ [활용 분야] ← AI 생성 (불릿)                  │
├─────────────────────────────────────────────┤
│ [기대 효과] ← AI 생성 (불릿)                  │
├─────────────────────────────────────────────┤
│ [시장 동향] ← 수동 입력 (선택)                │
├─────────────────────────────────────────────┤
│ [특허/권리 현황] ← KIPRIS                     │
├─────────────────────────────────────────────┤
│ [핵심 키워드] ← AI 생성 (태그)                │
├─────────────────────────────────────────────┤
│ [기술이전 문의] ← 제니스 담당자 연락처         │
└─────────────────────────────────────────────┘
```

### 6.2 표시 규칙

- 수동 입력 항목이 비어있으면 섹션 자체를 **숨김** (빈 섹션 방지)
- TRL은 KISTA 스타일 가로 타임라인 컴포넌트로 시각화
- 핵심 키워드는 클릭 시 같은 키워드 특허 필터링 (검색 연동)
- 모바일 대응 필수

## 7. 구현 단계 (권장 순서)

> **각 단계는 별도 Claude Code 세션으로 진행 권장** (맥락 분리·품질 확보)

### 단계 1: DB 스키마 조사 및 마이그레이션
- 현재 `patents` 테이블 구조 조사
- 추가 컬럼 마이그레이션 SQL 작성
- Supabase에서 직접 적용
- **산출물**: `migrations/001_add_smk_fields.sql`

### 단계 2: AI 생성 API 라우트
- `app/api/smk/generate/route.js` 신규 작성
- 입력: 특허번호 → 출력: JSON 7개 항목
- 프롬프트는 별도 파일 `lib/prompts/smk-generator.js` 로 분리
- **산출물**: API 엔드포인트 + 프롬프트 파일

### 단계 3: 기존 조회 플로우와 연결
- `app/page.jsx` 검색 성공 시 SMK 자동 생성 호출
- 로딩 스피너 단계에 "SMK 생성 중" 추가
- DB 저장 (status='ai_generated')

### 단계 4: 관리자 검토 UI
- `app/admin/page.jsx` 에 검토 모드 추가
- 항목별 편집, 병렬 비교, 재생성 버튼
- "검토 완료" → status='reviewed'

### 단계 5: 게시 플로우
- "플랫폼 등록" 버튼 → status='published'
- 공개 페이지 라우팅 (`app/patents/[id]/page.jsx` 신설 여부 검토)

### 단계 6: 공개 페이지 리디자인
- 6.1 레이아웃 구현
- 섹션 숨김 로직
- TRL 타임라인 컴포넌트
- 키워드 태그 필터링

### 단계 7: 품질·운영 개선
- AI 재생성 이력 기록
- 실패 복구 (재시도 큐)
- 관리자 대시보드 (검토 대기 목록)

## 8. 리스크 및 유의사항

### 8.1 기술적 리스크

| 리스크 | 완화 방안 |
|---|---|
| AI 응답이 JSON 스키마 벗어남 | `response_format: json` 사용 또는 zod 검증 |
| Claude API 호출 실패·타임아웃 | 재시도 로직 + 관리자에게 실패 알림 |
| 명세서 XML 파싱 실패 | 기존 `figureDescsFromXml` 로직 재사용 + 예외 처리 |
| 비용 폭증 (특허당 3~5K 토큰) | 관리자가 명시적으로 "생성" 버튼 눌러야 호출되도록 |

### 8.2 법무·윤리적 리스크

| 리스크 | 완화 방안 |
|---|---|
| AI 생성 결과가 사실과 다름 | 관리자 검토를 **필수 단계**로 강제 (검토 없이 게시 불가) |
| 특허권자 동의 없는 가공 | 기술이전 협약 체결 후에만 등록 (플랫폼 운영 약관에 명시) |
| 시장 규모 등 정량 데이터 오류 | AI 생성 항목에서 완전 배제 (4.2 원칙 2) |
| 경쟁사 특허 분석용 악용 가능성 | 검색 결과는 로그인 + 목적 확인 필요 (별도 과제) |

### 8.3 운영 리스크

| 리스크 | 완화 방안 |
|---|---|
| 검토 병목 (담당자 업무량 폭증) | 우선순위 큐 + 관리자 여러 명 권한 분리 |
| 생성 품질 편차 | 프롬프트 버전 관리 + A/B 테스트 기반 개선 |
